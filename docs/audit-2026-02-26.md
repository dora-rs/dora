Adora is a well-architected Rust robotics framework with genuinely effective zero-copy shared memory for large messages, clean four-layer design, and strong error handling patterns. Since the prior audit (2026-02-18), all 4 critical findings and 10 of 12 high findings have been fixed.

This audit expands scope to include **performance analysis** and **test coverage assessment** -- two dimensions not covered previously. It also identifies new findings across all categories.

## New Findings: Performance

### H-P1. 100 Million Event Channel Buffer (HIGH)

**Location**: `apis/rust/node/src/event_stream/mod.rs:235`

```rust
flume::bounded(100_000_000)
```

Effectively unbounded. A slow consumer with large `DataMessage` events could consume gigabytes of memory before any backpressure signal. Combined with `UnboundedSender` for daemon subscribe channels (`daemon/lib.rs:3016`), there is zero end-to-end backpressure.

**Impact**: Memory exhaustion under load, potential OOM kill of the entire daemon process.

**Fix options**:
1. **Reduce to 10,000** with proper backpressure (back-propagate to daemon)
2. **Adopt latest-data semantics** like the operator channel already does (`runtime/operator/channel.rs:101-127`), dropping oldest events when queue fills
3. **Make configurable** per-node in YAML with a sane default (e.g., 1000)

### H-P2. Data Cloned Per Subscriber in Daemon (HIGH)

**Location**: `binaries/daemon/src/lib.rs:2688-2689`

```rust
metadata.clone()  // full copy of Metadata struct
data.clone()      // for small messages: full copy of AVec data bytes
```

Called once per local subscriber. For a node output with N local subscribers receiving small messages (<4KB), this produces N full copies of the message data.

**Impact**: Linear scaling cost with fan-out degree. For high-frequency, multi-subscriber dataflows, this dominates CPU time.

**Fix**: Wrap in `Arc<DataMessage>` to share across subscribers. Only clone the `Arc` pointer (1 atomic increment per subscriber instead of full memcpy).

### M-P1. Per-Message Allocations on Hot Path (MEDIUM)

Multiple allocations per message that could be pooled or eliminated:

| Allocation | Location | Frequency |
|-----------|----------|-----------|
| `oneshot::channel()` per shmem op | `daemon/node_communication/shmem.rs:63,76` | Every shmem message (2 allocs) |
| `String` for shmem ID | `apis/rust/node/src/node/mod.rs:999` | Every large message send |
| `Vec<u8>` for bincode serialize | `shared-memory-server/src/channel.rs:89` | Every control message |
| `Vec<u8>` for TCP receive buffer | `daemon/src/socket_stream_utils.rs:51` | Every TCP message |
| `vec![0; reply_len]` TCP receive | `daemon/node_communication/tcp.rs:107` | Every TCP message |

**Fix**: Pool allocations. Reuse buffers across messages. For shmem IDs, consider a fixed-size `[u8; 32]` instead of `String`.

### M-P2. Sequential Daemon Main Loop (MEDIUM)

**Location**: `binaries/daemon/src/lib.rs:429-647`

All node events are processed sequentially in a single-threaded async event loop. The code already includes a 100ms slow-handler warning (`lib.rs:624`), confirming this is a known bottleneck.

A slow handler (e.g., Zenoh publish to a congested network) stalls all nodes on that daemon.

**Fix options**:
1. **Spawn handlers**: Use `tokio::spawn` for potentially slow operations (Zenoh, shmem allocation)
2. **Shard by dataflow**: Run one event loop per dataflow, isolating failures
3. **Accept the tradeoff**: Document this as a design choice (simpler reasoning, no lock contention)

### M-P3. No Release Profile Optimizations (MEDIUM)

**Location**: `Cargo.toml` (workspace root)

```toml
[profile.dist]
inherits = "release"
lto = "thin"
```

Only the `dist` profile has thin LTO. The default `release` profile has no optimizations beyond Rust defaults. The benchmark example uses `--release` (not `--profile dist`), so LTO isn't active during benchmarking.

Missing optimizations:
- `codegen-units = 1` (better optimization, slower compile)
- `strip = true` (smaller binary)
- Full LTO for `dist` profile (currently only thin)

**Fix**: Add `codegen-units = 1` and `lto = "fat"` to `dist` profile. Keep `release` profile fast for development.

### M-P4. Remote Data Path Copies Shared Memory to Vec (MEDIUM)

**Location**: `binaries/daemon/src/lib.rs:2770-2774`

For cross-machine (Zenoh) delivery of shared memory data, the daemon maps the shmem region and copies it into an `AVec`:

```rust
// Full copy from shared memory to AVec for Zenoh transmission
```

This is inherent to the distributed architecture (can't share OS shmem across machines), but means the "zero-copy" claim only holds for single-machine deployments.

**Fix**: Document this clearly. Consider Zenoh's built-in shared memory transport for co-located daemons.

### L-P1. Request-Reply Event Model Adds Latency (LOW)

**Location**: `apis/rust/node/src/event_stream/mod.rs`

Nodes use a request-reply pattern (`NextEvent` -> `NextEvents`): the node asks the daemon for the next events, the daemon responds. This is sequential per node -- a node cannot receive a new batch until it processes the previous one and asks again.

A push model would reduce latency by eliminating the round-trip.

**Impact**: Adds ~1 RTT of latency per event batch. Acceptable for most robotics workloads.

### L-P2. Benchmark Latency Results (Informational)

From `examples/benchmark/out/` (latest run):

| Message Size | Latency | Transport |
|-------------|---------|-----------|
| 0 bytes | 855 us | TCP |
| 64 bytes | 433 us | TCP |
| 512 bytes | 562 us | TCP |
| 2048 bytes | 777 us | TCP |
| 4096 bytes | 467 us | Shared memory |
| 16 KB | 512 us | Shared memory |
| 40 KB | 530 us | Shared memory |
| 400 KB | 526 us | Shared memory |
| 4 MB | 412 us | Shared memory |

**Key observation**: Latency is flat from 4KB to 4MB (412-530us), confirming zero-copy works on the receiver side. The 4KB shmem threshold is well-chosen.

### What Works Well (Performance)

- **Shared memory caching**: Up to 20 regions cached per node (`MAX_CACHE_SIZE = 20`), avoiding repeated syscalls
- **Arrow `Buffer::from_custom_allocation`**: Receiver wraps shmem pointer directly -- genuinely zero-copy
- **No locks in daemon**: Single-threaded async eliminates lock contention
- **128-byte aligned allocations**: `AVec<u8, ConstAlign<128>>` for cache-line alignment
- **`TCP_NODELAY`**: Set on all TCP connections
- **Operator queue management**: Proper latest-data semantics with `drop_oldest_inputs`

## New Findings: Testing

### H-T1. Zero Test Coverage on Performance-Critical Crates (HIGH)

| Crate | Lines | Tests | Risk |
|-------|-------|-------|------|
| `shared-memory-server` | 458 | 0 | Core zero-copy differentiator |
| `adora-runtime` | 336 | 0 | Operator execution runtime |
| `adora-operator-api` | 449 | 0 | Key user-facing API |
| `daemon/src/lib.rs` | 5,379 | 6 | Most complex component |

The shared memory server -- the foundation of the 10-17x latency claim -- has zero tests. The 6 daemon tests only cover input close/break/recover state transitions; spawning, routing, and Zenoh logic are untested at the unit level.

**Fix priority**:
1. `shared-memory-server`: buffer allocation, read/write round-trips, edge cases (max-size, zero-size)
2. `adora-runtime`: operator loading, event routing, error propagation
3. `daemon/lib.rs`: message routing, spawn logic, error paths
4. `adora-operator-api`: operator lifecycle, input/output handling

### M-T1. No CI Coverage Reporting (MEDIUM)

No coverage tool (`cargo-tarpaulin`, `cargo-llvm-cov`) is configured. Coverage claims are unverifiable.

**Fix**: Add `cargo-llvm-cov` to CI. Target 80% for library crates per project guidelines.

### M-T2. CI Runs on Ubuntu Only (MEDIUM)

**Location**: `.github/workflows/ci.yml`

CI runs exclusively on `ubuntu-latest` despite the project supporting macOS and Windows. Platform-specific issues in process spawning, shared memory, and path handling could go undetected.

**Fix**: Add macOS CI job at minimum. Windows can be added later (more complex).

### M-T3. Smoke Tests Only Check Exit Codes (MEDIUM)

**Location**: `tests/example-smoke.rs`

Most smoke tests assert only that the process exits successfully. They don't verify that:
- The dataflow produced correct output
- Shared memory was actually used for large messages
- Specific nodes communicated properly

A dataflow that silently drops half its messages would pass all smoke tests.

**Fix**: Add output content verification to at least the benchmark and rust-dataflow smoke tests.

### M-T4. No Benchmarks for Regression Testing (MEDIUM)

No `criterion` or `divan` benchmarks exist. The `examples/benchmark/` is a manual dataflow, not an automated regression test. Performance improvements or regressions cannot be detected in CI.

**Fix**: Add `criterion` micro-benchmarks for:
- Shmem channel round-trip latency
- Arrow buffer pack/unpack
- Message routing throughput (daemon hot path)

### L-T1. Weak Assertion in `cli_stop_nonexistent` (LOW)

**Location**: `tests/ws-cli-e2e.rs`

```rust
msg.contains(&fake_uuid.to_string())
    || msg.to_lowercase().contains("not found")
    || !msg.is_empty()  // <-- always true if error is non-empty
```

The `!msg.is_empty()` arm makes this assertion trivially true for any non-empty error string.

### L-T2. Test Isolation Relies on `thread::sleep` (LOW)

**Location**: `tests/example-smoke.rs`

The `cleanup_stale()` function uses `thread::sleep(500ms)` for synchronization. Smoke tests require `--test-threads=1`. Fragile under CI load.

### L-T3. No Property-Based Testing for Arrow Conversions (LOW)

The `adora-arrow-convert` crate has 27 round-trip tests but no property-based testing with `proptest` or `quickcheck`. Fuzz-like coverage would catch edge cases in type conversion.

## New Findings: Architecture

### H-A5. No Coordinator-Daemon State Reconciliation on Restart (HIGH)

**Location**: `binaries/coordinator/src/lib.rs:156-187`

On coordinator restart, all previously-running dataflows are marked as `Failed`:

```rust
record.status = StoreDataflowStatus::Failed {
    error: "coordinator restarted".into(),
};
```

Comment: "full reconciliation with daemons is Phase 2 work."

A coordinator crash terminates all running dataflows. This is a significant availability gap for production robotics.

**Fix**: Implement reconnect protocol where daemons report live dataflows and coordinator reconciles state.

### M-A10. `Node` Struct Uses 5 Mutually Exclusive `Option` Fields (MEDIUM)

**Location**: `libraries/message/src/descriptor.rs:142-617`

The `Node` struct has 24 optional fields. Five represent mutually exclusive node kinds (`path`, `operators`, `custom`, `operator`, `ros2`), checked at runtime:

```rust
fn kind(&self) -> eyre::Result<NodeKind<'_>> {
    match (&self.path, &self.operators, &self.custom, &self.operator, &self.ros2) {
        (None, None, None, None, None) => bail!("..."),
        _ => bail!("multiple exclusive fields set"),
    }
}
```

Invalid states are representable. A proper `NodeKind` enum would make this a parse-time error.

**Fix**: Refactor to discriminated union:
```rust
pub struct Node {
    pub id: NodeId,
    pub name: Option<String>,
    pub env: Option<BTreeMap<String, EnvValue>>,
    pub kind: NodeKind,
}

pub enum NodeKind {
    Process(ProcessNode),
    Runtime(RuntimeNode),
    Operator(SingleOperatorDefinition),
    Ros2Bridge(Ros2BridgeConfig),
}
```

**Trade-off**: This changes the YAML schema. Users would need to nest under a `kind:` key. Could use `#[serde(untagged)]` to preserve the flat format, at the cost of worse error messages.

### M-A11. Full Dataflow Descriptor Embedded in Node Config Env Var (MEDIUM)

**Location**: `libraries/message/src/daemon_to_node.rs:26`

```rust
pub struct NodeConfig {
    pub dataflow_descriptor: serde_yaml::Value,  // entire YAML
    // ...
}
```

The complete dataflow descriptor is serialized into `ADORA_NODE_CONFIG` env var. Environment variables have OS-level size limits (128KB-2MB). Large dataflows could exceed this.

**Fix**: Remove `dataflow_descriptor` from `NodeConfig`. The node can fetch it lazily via the existing `DaemonRequest::NodeConfig` mechanism.

### M-A12. `DaemonId` Uses Random UUID (v4) While Others Use Time-Ordered (v7) (MEDIUM)

**Location**: `libraries/message/src/common.rs:329`

`SessionId` and `BuildId` use `Uuid::new_v7` (time-ordered, sortable). `DaemonId` uses `Uuid::new_v4` (random). Inconsistent and loses sortability in logs.

Additionally, `DaemonId` is regenerated on every daemon restart, making coordinator reconciliation impossible.

**Fix**: Use `v7` for consistency. Consider persisting `DaemonId` to disk for stable identity across restarts.

### M-A13. Errors Serialized to `String` at Coordinator Boundary (MEDIUM)

**Location**: `libraries/message/src/daemon_to_coordinator.rs`

```rust
pub enum DaemonEvent {
    BuildResult { result: Result<(), String> },
    SpawnResult { result: Result<(), String> },
}
```

The coordinator cannot distinguish error categories (transient vs permanent) without string parsing.

**Fix**: Replace `String` with a structured `DaemonError` enum for programmatic error handling.

## New Findings: Security

### M-S9. Shell Node Env Var Opt-in Is Inherited (MEDIUM)

**Location**: `binaries/daemon/src/spawn/command.rs:22-41`

`ADORA_ALLOW_SHELL_NODES=true` is checked via env var, which is silently inherited from parent processes. A CI system that exports this globally enables shell execution for all untrusted YAML dataflows.

**Fix**: Use CLI flag `--allow-shell-nodes` instead. Flags require explicit intent per invocation.

### M-S10. `args` Field Split on Whitespace -- No Quoting (MEDIUM)

**Location**: `binaries/daemon/src/spawn/command.rs:105-107`

```rust
cmd = cmd.args(args.split_ascii_whitespace());
```

Arguments with spaces cannot be passed correctly.

**Fix**: Accept `args` as `Vec<String>` in YAML, or use shell-quote parsing library.

### L-S1. No CORS on Coordinator WebSocket (LOW)

**Location**: `binaries/coordinator/src/ws_server.rs:26-55`

`tower-http` with `cors` feature is in `Cargo.toml` but no `CorsLayer` is applied.

### L-S2. `rand_jitter_millis()` Uses System Time as Randomness (LOW)

**Location**: `binaries/daemon/src/coordinator.rs:259-264`

```rust
fn rand_jitter_millis() -> u64 {
    SystemTime::now().duration_since(UNIX_EPOCH).unwrap_or_default().subsec_nanos() as u64
}
```

Not cryptographically random. Daemons started simultaneously will have correlated jitter, defeating thundering-herd protection.

**Fix**: Use `rand::thread_rng()`.

### L-S3. TCP Listener Accepts Unlimited Concurrent Connections (LOW)

**Location**: `binaries/daemon/src/node_communication/tcp.rs:30-50`

No connection limit. A rogue process can exhaust file descriptors.

**Fix**: Track active connections, reject beyond threshold (e.g., 16 per node).

## What Works Well

### Architecture
- Clean four-layer pipeline (CLI -> Coordinator -> Daemon -> Nodes)
- `adora run` collapses to single-process for development -- elegant optimization
- Independent `adora-message` versioning (0.7.0 vs 0.4.1) for wire protocol stability
- Extension pattern (`libraries/extensions/`) keeps ROS2, telemetry, download opt-in
- `CoordinatorStore` trait enables pluggable persistence backends

### Code Quality
- Consistent `eyre` + `WrapErr` throughout, producing excellent error messages
- `#[serde(deny_unknown_fields)]` on all YAML types catches typos at parse time
- `schemars::JsonSchema` on `Descriptor` enables IDE completion
- `CachedResult` pattern for coalescing multiple waiters (coordinator)
- `#[must_use]` on `FinishDataflowWhen`
- `catch_unwind` wrapping of operator event loops prevents panic propagation
- Version compatibility checked at registration time

### Security
- All 4 prior critical findings fixed
- `MAX_MESSAGE_BYTES = 256 MiB` enforced on all TCP paths with 30s timeout
- Download filename sanitization with `Path::file_name()`, null-byte rejection
- Daemon binds to loopback (`127.0.0.1`) by default
- Daemon ID cross-check prevents impersonation on existing connections
- No hardcoded credentials in source
- No `transmute` or `MaybeUninit` -- common UB sources absent

### Performance
- Genuinely zero-copy for large messages via shared memory + Arrow
- Flat latency from 4KB to 4MB (confirmed by benchmarks)
- No locks in daemon (single-threaded async)
- 128-byte aligned allocations for Arrow data
- `TCP_NODELAY` on all connections
- Shared memory region caching (up to 20 per node)
- Operator queue management with latest-data semantics

### Testing
- Five-tier test hierarchy (unit, integration, smoke, E2E, fault tolerance)
- Excellent testing guide documentation
- First-class integration testing framework in node API
- Coordinator tests use real WebSocket connections on random ports
- `Once` guards for shared build steps in smoke tests
