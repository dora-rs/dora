# Adora Deep Technical Audit Report

**Date**: 2026-02-18
**Last updated**: 2026-02-20
**Prior audit**: 2026-02-16
**Audited by**: Claude Code (multi-agent: architect, security-reviewer, code-reviewer, unsafe-reviewer)
**Scope**: Full workspace -- architecture, security, code quality, unsafe code

## Overall Assessment

Adora is a well-structured, production-intent Rust robotics framework with excellent core design decisions (zero-copy shared memory, Arrow format, UHLC distributed clocks). The architecture is sound at a high level. This audit uncovered **4 critical security vulnerabilities**, **12 high-severity issues**, and numerous medium/low findings. As of 2026-02-20: **all 4 critical findings are fixed** (C1-C5), and **4 high findings are resolved** (H-A2, H-A3 partial, H-A4, H-Q2). The remaining high-priority items are C2 (no auth), H-S1-S6 (security), H-U1-U6 (unsafe), and H-Q1/Q3-Q5 (quality).

---

## What's Done Well

- **Zero-copy architecture**: Shared memory for >4KB messages + Apache Arrow columnar format
- **Clean layering**: CLI -> Coordinator -> Daemon -> Nodes/Operators is well-separated
- **UHLC (Hybrid Logical Clock)**: Proper distributed causality tracking
- **Multi-language support**: Rust/Python/C/C++ APIs with clean FFI boundaries
- **Consistent error chains**: `eyre` + `.wrap_err()` / `.context()` throughout
- **Atomic ordering discipline**: Release/Acquire in `ShmemChannel` is correct
- **No `transmute` or `MaybeUninit`** anywhere -- common UB sources are absent
- **No hardcoded credentials** found in source
- **`cxx::bridge`** for C++ interop provides compile-time safety
- **Version compatibility** verified at daemon/node registration handshake
- **Coordinator heartbeat watchdog** disconnects stale daemons after 30s
- **`parse_log`** validates input size (64 KB limit) -- good boundary hardening
- **`#[must_use]`** on `FinishDataflowWhen` enforces correct handling
- **`catch_unwind`** wrapping of operator event loops prevents panic propagation

## Critical Findings (P0)

### C1. Shell Command Injection (RCE)

**Location**: `binaries/daemon/src/spawn/command.rs:21-28`
**Type**: CWE-78 OS Command Injection
**Status**: **Fixed** (`6e3278b`, 2026-02-20) -- gated behind `ADORA_ALLOW_SHELL_NODES=true` env var with warning log.

When a node's `path` is `"shell"`, the `args` field is passed verbatim to `sh -c`. Any user who can supply a dataflow YAML gets arbitrary command execution as the daemon user.

```rust
SHELL_SOURCE => {
    let cmd = Command::new("sh");
    cmd.args(["-c", &node.args.clone().unwrap_or_default()])
}
```

**Original fix**: Remove `SHELL_SOURCE` or gate behind `--allow-shell-nodes` CLI flag + build feature.
**Residual risk**: Env-var opt-in is weaker than a CLI flag since env vars can be inherited silently from a parent process.

### C2. No Authentication on Any Network Interface

**Location**: `coordinator/src/control.rs`, `listener.rs`, `daemon/src/node_communication/tcp.rs`
**Type**: CWE-306 Missing Authentication for Critical Function

All TCP interfaces (CLI<->Coordinator, Coordinator<->Daemon, Daemon<->Node) accept connections with zero authentication. The control port binds `0.0.0.0` by default. Any process on the network can kill dataflows, launch arbitrary nodes, or impersonate a daemon.

**Fix**: Add shared token at minimum; mTLS for production. Restrict binding to loopback by default.

### C3. Unbounded Memory Allocation via Network (DoS)

**Location** (4 instances):
- `coordinator/src/tcp_utils.rs:15-22`
- `communication-layer/request-reply/src/tcp.rs:139-144`
- `daemon/src/socket_stream_utils.rs:14-24`
- `apis/rust/node/src/daemon_connection/tcp.rs:78-83`

**Type**: CWE-789 Memory Allocation with Excessive Size Value
**Status**: **Fixed** (`6e3278b` + `0d3427d`, 2026-02-20) -- `MAX_MESSAGE_BYTES = 256 MiB` constant in `adora-message`; all 4 TCP paths reject oversized frames. A 30-second `TCP_READ_TIMEOUT` was also added.

Every TCP receive path reads an 8-byte LE length prefix then allocates `Vec<u8>` of that size with no bound check. A single crafted packet causes OOM crash.

```rust
let reply_len = u64::from_le_bytes(raw) as usize;  // Up to 18 EiB
let mut reply = vec![0; reply_len];                  // Instant OOM
```

**Original fix**: Add `MAX_MESSAGE_BYTES` constant (e.g., 256 MiB) and reject before allocating.

### C4. Path Traversal via `Content-Disposition` in File Download

**Location**: `libraries/extensions/download/src/lib.rs:7-44`
**Type**: CWE-22 Path Traversal
**Status**: **Fixed** (`6e3278b`, 2026-02-20) -- `Path::file_name()` sanitization, null-byte rejection, and 255-byte length cap applied.

Downloaded filenames from HTTP headers are joined to `target_dir` unsanitized. A malicious server returning `filename=../../bin/evil` writes outside the intended directory.

```rust
fn get_filename(response: &reqwest::Response) -> Option<String> {
    // ...
    return Some(name.trim_matches('"').to_string());  // No sanitization
}
let path = target_dir.join(filename);  // "../../evil" traverses up
```

**Original fix**: Use `Path::new(&filename).file_name()` only; reject directory separators. Add SHA-256 checksum field to YAML spec (tracked as M-S3).

### C5. `Timestamped::serialize()` Panics in Hot Data Path

**Location**: `libraries/message/src/common.rs:255`
**Status**: **Fixed** (`6e3278b`, 2026-02-20) -- returns `eyre::Result<Vec<u8>>` with `.wrap_err()`; all callers updated.

```rust
pub fn serialize(&self) -> Vec<u8> {
    bincode::serialize(self).unwrap()  // panics on every inter-daemon message failure
}
```

**Original fix**: Change to `-> eyre::Result<Vec<u8>>` and update all callers.

## High Findings (P1)

### Architecture

#### H-A1. CLI Embeds Daemon + Coordinator + Runtime -- BY DESIGN

**Location**: `binaries/cli/Cargo.toml:47-49`
**Status**: Reclassified as **by design** (2026-02-20)

All three binary crates are included as libraries in `adora-cli`. This is intentional: `adora run` calls `Daemon::run_dataflow()` in-process for single-machine execution without spawning separate coordinator/daemon processes. The distributed path (`adora coordinator` + `adora daemon` + `adora start`) uses them as separate subcommands. Both paths share `Daemon::run_general()` as the core runtime, ensuring identical behavior. The binary size cost is justified by the developer UX of `adora run dataflow.yml` working out of the box.

**No fix needed.**

#### H-A2. Python Node API Depends on CLI

**Location**: `apis/python/node/Cargo.toml:30-31`
**Status**: **Fixed** (`e35a115`, 2026-02-20) -- CLI functions extracted to new `apis/python/cli/` crate (`adora-cli-api-python`). Python node API no longer depends on `adora-cli`, `adora-runtime`, or `adora-download`.

A language-binding API crate depending on the top-level CLI binary crate is a significant layering inversion. This pulls the entire CLI dependency tree into every Python build.

**Original fix**: Python node API should only depend on `adora-node-api` and `adora-message`.

#### H-A3. Library Crates Use `eyre` Exclusively -- No Typed Errors

**Status**: **Partially fixed** (`e35a115`, 2026-02-20) -- `NodeError` enum added to `adora-node-api` with variants `Init`, `Connection`, `Output`, `Data`, `Internal(#[from] eyre::Report)`. All public functions return `NodeResult<T>`. The `Internal` catch-all means most `?`-propagated errors still land in `Internal`; explicit variants are used only at direct construction sites. `adora-message` is not yet converted.

All library crates (`adora-message`, `adora-core`, `adora-node-api`, `shared-memory-server`) use `eyre::Result` for all error propagation. Consumers cannot programmatically match on specific error variants.

**Original fix**: Introduce `thiserror`-derived error types in at least `adora-node-api` and `adora-message`.

#### H-A4. Coordinator is a Single 1700-Line File

**Location**: `binaries/coordinator/src/lib.rs`
**Status**: **Fixed** (`e35a115`, 2026-02-20) -- split into `state.rs` (~180 lines), `events.rs` (~110 lines), `handlers.rs` (~580 lines). `lib.rs` reduced to ~900 lines. The `start_inner` event-dispatch loop remains in `lib.rs` but all handler logic is extracted.

`start_inner` alone is ~850 lines handling 15+ event variants. Untestable in current form.

**Original fix**: Split into `state.rs`, `handlers/control.rs`, `handlers/daemon.rs`, `handlers/dataflow.rs`.

### Security

#### H-S1. `todo!()` Panic in Active Runtime Path

**Location**: `binaries/runtime/src/lib.rs:177`

The `StopAll` operator event handler panics the runtime.

**Fix**: Implement the feature or return an `Err`.

#### H-S2. Shared Memory World-Readable (0666)

**Location**: `daemon/src/node_communication/mod.rs:80-160`

POSIX shared memory defaults to mode `0666`. Any process on the machine can read/write shmem regions, inject data, cause memory corruption.

**Fix**: Set permissions to `0o600` after creation.

#### H-S3. `assert!` in Shmem Channel Crashes Daemon

**Location**: `shared-memory-server/src/channel.rs:94-95, 151-153`

Malformed input from a node panics the daemon process.

```rust
assert!(msg.len() <= self.memory.len() - self.data_offset);  // PANICS
```

**Fix**: Replace `assert!` with `eyre::bail!` or `Err(...)` returns.

#### H-S4. Environment Variable Injection via YAML `env` Field

**Location**: `daemon/src/spawn/spawner.rs:138-149`
**Status**: NEW

No filtering of dangerous variables like `LD_PRELOAD`, `DYLD_INSERT_LIBRARIES`, `PYTHONPATH`, `PATH`. An attacker who supplies a YAML file can hijack the node's dynamic linker.

**Fix**: Deny a blocklist of dangerous variables or use `env_clear()` with an allowlist.

#### H-S5. `node_id` Path Traversal in Log/Socket Paths

**Location**: `daemon/src/log.rs:24-38`, `daemon/src/node_communication/mod.rs:169`
**Status**: NEW

Node IDs from YAML are injected into file paths unsanitized. A `NodeId` containing `../../../etc/cron.d/backdoor` writes to sensitive locations.

**Fix**: Validate `NodeId` at parse time: only `[a-zA-Z0-9_-]` allowed.

#### H-S6. `raw_sync_2` Supply Chain Risk

**Location**: `shared-memory-server/Cargo.toml:20`

Core IPC depends on an unvetted personal crate fork handling raw OS synchronization primitives.

**Fix**: Pin exact version; evaluate migrating to maintained alternative; track upstream PR.

### Unsafe Code

#### H-U1. `align_offset` Return Not Checked for `usize::MAX`

**Location**: `shared-memory-server/src/channel.rs:39-52`

`align_offset()` returns `usize::MAX` when alignment is impossible. The code uses the result without checking, causing `wrapping_add(usize::MAX)` -- writes to arbitrary memory.

**Fix**: `assert_ne!(align, usize::MAX, "alignment impossible")` after every `align_offset()` call.

#### H-U2. Null Pointer UB in `FFISeq::deref`

**Location**: `ros2-bridge/src/_core/sequence.rs:65-71`

When `size == 0`, `self.data` may be null. `slice::from_raw_parts(null, 0)` is UB.

```rust
impl<T> Deref for FFISeq<T> {
    fn deref(&self) -> &[T] {
        unsafe { std::slice::from_raw_parts(self.data, self.len()) }
    }
}
```

**Fix**: Guard with `if self.size == 0 { return &[]; }`.

#### H-U3. No Null Check in C Node API `try_send_output`

**Location**: `apis/c/node/src/lib.rs:261-271`

`id_ptr` and `data_ptr` are not null-checked before `slice::from_raw_parts`. A C caller passing `NULL, 0` causes UB. Contrast with `try_log` which does null-check.

**Fix**: Add null-pointer guards matching the pattern in `try_log`.

#### H-U4. `MappedInputData::len` Not Validated Against Region Size

**Location**: `apis/rust/node/src/event_stream/data_conversion.rs:71-77`

`self.len` is provided by the daemon. If it exceeds the shmem region size, `deref` panics on the slice bounds check (not UB in safe Rust, but crashes the node).

**Fix**: Assert `len <= memory.len()` at `map()` time.

#### H-U5. `CStr::from_ptr` Ignores `size` Field

**Location**: `ros2-bridge/src/_core/string.rs:90-97`

`CStr::from_ptr` scans until null byte, ignoring `self.size`. A missing null terminator reads beyond the buffer.

**Fix**: Use `CStr::from_bytes_with_nul` with a bounded slice.

#### H-U6. `#[allow(clippy::missing_safety_doc)]` Crate-Wide in 5 Crates

**Location**: `shared-memory-server/src/lib.rs:1`, `ros2-bridge/src/lib.rs:1`, `operator/src/lib.rs:19`, `operator/types/src/lib.rs:3`, `shared-memory-server/src/channel.rs:22`

All `unsafe fn` items lack `# Safety` sections. Future maintainers cannot reason about correctness.

**Fix**: Remove the allows; write `# Safety` docs for every `unsafe fn`.

### Code Quality

#### H-Q1. `Daemon` God-Struct (23 Fields, 4000 Lines)

**Location**: `binaries/daemon/src/lib.rs`

Mixes coordinator connections, session state, build state, node communication, metrics, git management, and logging. `run_inner` is a 200-line match with 13 arms. Untestable in isolation.

**Fix**: Decompose into `DaemonConfig`, `SessionState`, `NodeManager`, `BuildManager`.

#### H-Q2. `Uuid::parse_str().unwrap()` on Untrusted Log Input

**Location**: `libraries/message/src/common.rs:53, 56, 67, 72`
**Status**: **Fixed** (`6e3278b`, 2026-02-20) -- all four `Uuid::parse_str()` calls now use `.ok()` with `.map()`/`.and_then()`.

Inside `From<LogMessageHelper> for LogMessage`, reached when parsing external log JSON. A corrupted log message crashes the coordinator.

**Original fix**: Use `.ok()` + `filter_map` or return a `Result`.

#### H-Q3. `LazyLock<Runtime>` Panics at First Python API Call

**Location**: `apis/python/node/src/lib.rs:25-32`

Runtime creation failure (OOM, thread limit) is an unrecoverable panic with no way for the Python caller to catch it.

**Fix**: Convert to a function returning `PyResult<&'static Runtime>`.

#### H-Q4. `parse_dependencies` Uses `panic!` in Library Code

**Location**: `ros2-bridge/msg-gen/src/parser/package.rs:192`

A malformed ROS2 package XML triggers an unrecoverable crash instead of an error.

**Fix**: Return `Err(...)`.

#### H-Q5. `init_tracing` Panics on Bad OTLP Endpoint

**Location**: `libraries/extensions/telemetry/tracing/src/telemetry.rs:47`

`.build().unwrap()` on the OTLP exporter builder. A bad `ADORA_OTLP_ENDPOINT` crashes the node.

**Fix**: Return `Result`.
